BUSINESS CONTEXT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
This is a multi-site Veeam Backup & Replication environment.
Multiple VBR servers (vbr_server) manage backup jobs across VMs, physical agents, and NAS/file servers.
Each backup job runs on a schedule and protects one or more objects (VMs, agents, file shares).
The database tracks failures, protection status, repository usage, restore points, and job performance.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TABLE: failed_jobs_daily
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Purpose: One row per object that failed in a backup job on a given date.
Every row in this table represents a failure — there is NO status column here.

Columns:
  backup_date          DATE       — the calendar date the failure occurred
  vbr_server           TEXT       — which VBR server ran the job (e.g. VBR1, VBR2, VBR3)
  job_name             TEXT       — name of the backup job (e.g. Prod_VM_Job, Linux_VM_Job)
  failed_object_name   TEXT       — the specific VM, agent, or file server that failed (e.g. WApp01, WAgt03, NAS-SRV01)
  failure_message      TEXT       — the actual error text explaining why it failed

Key facts:
- The same object can fail on multiple days — each day is a separate row.
- The same job can appear multiple times if multiple objects failed within it.
- To find unique objects that failed: use DISTINCT on failed_object_name.
- For recency: backup_date = CURRENT_DATE means today, >= CURRENT_DATE-1 means last 24h, >= CURRENT_DATE-7 means last week.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TABLE: protected_vms
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Purpose: Catalogue of every object currently under backup protection.

Columns:
  object_name              TEXT        — name of the protected object (VM, agent, or file server)
  object_type              TEXT        — type: 'VM', 'Agent', or 'File Server'
  job_name                 TEXT        — which backup job protects this object
  vbr_server               TEXT        — which VBR server manages this job
  last_successful_backup   TIMESTAMP   — when this object last had a successful backup
  active_restore_points    INT         — number of restore points currently available
  size_gb                  NUMERIC     — size of the protected object in GB

CRITICAL: The same object_name appears multiple times if it is included in more than one job.
  - ALWAYS use COUNT(DISTINCT object_name) when counting protected servers/VMs.
  - ALWAYS use SELECT DISTINCT object_name when listing protected servers/VMs.
  - Using COUNT(*) or COUNT(object_name) returns inflated numbers.

Columns:
  job_name             TEXT       — name of the backup job
  backup_type          TEXT       — type: 'VM Backup', 'Agent Backup', 'File Backup', etc.
  vbr_server           TEXT       — which VBR server ran the job

  end_time             TIMESTAMP  — when the job finished

Important: This is the ONLY table with a status column. Do not filter by status in any other table.

LONG-RUNNING JOBS (use job_sessions, not a separate table)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Thresholds: Warning = over 6 hours (360 min), Critical = 24 hours or more (1440 min).
job_sessions uses start_time (TIMESTAMP) for time filters, not backup_date.
  "Recent" or "last 24h" → start_time >= NOW() - INTERVAL '1 day'
  "Today"                → start_time >= CURRENT_DATE

SLA DEFINITION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
SLA = percentage of protected objects that have a successful backup within the reporting window.

Source of truth: protected_vms.last_successful_backup (TIMESTAMP).
A job that fails but is retried and succeeds gets its last_successful_backup updated — it IS in SLA.
A job that keeps failing with no successful retry has an old last_successful_backup — it is an SLA miss.

SLA has two calculation modes depending on time period:

TODAY's SLA — use failed_jobs_daily:
  total_protected = COUNT(DISTINCT object_name) FROM protected_vms
  failed_today    = COUNT(DISTINCT failed_object_name) FROM failed_jobs_daily WHERE backup_date = CURRENT_DATE
  in_sla          = total_protected - failed_today
  sla_pct         = ROUND(100.0 * in_sla / NULLIF(total_protected, 0), 1)

WEEKLY / MONTHLY SLA — use protected_vms.last_successful_backup:
  in_sla   = DISTINCT objects where last_successful_backup >= CURRENT_DATE - N
  sla_miss = DISTINCT objects where last_successful_backup < CURRENT_DATE - N OR last_successful_backup IS NULL
  sla_pct  = ROUND(100.0 * in_sla / NULLIF(total_protected, 0), 1)

SLA query must return exactly ONE aggregate row. Columns: total_protected, in_sla, sla_missed, sla_pct.
Never GROUP BY object — the entire result is a single summary.
Time period: "today" = use failed_jobs_daily with backup_date = CURRENT_DATE.
             "this week" or default = last_successful_backup >= CURRENT_DATE - 7.
             "this month" = last_successful_backup >= CURRENT_DATE - 30.

STALE / OVERDUE BACKUPS — different from SLA:
When user asks "which objects haven't been backed up in X days", "overdue backups", "stale backups",
or "objects failing for multiple days" — query protected_vms for objects where
last_successful_backup < CURRENT_DATE - X, returning object_name, object_type, last_successful_backup.
This reveals objects like a NAS that succeeded 4 days ago but has been failing since — week SLA
counts it as OK (within 7 days) but it is operationally at risk.

IMPORTANT — PostgreSQL integer division: COUNT() returns BIGINT. Always multiply by 100.0 FIRST:
  Correct:   ROUND(100.0 * COUNT(CASE WHEN ... THEN object_name END) / NULLIF(COUNT(DISTINCT object_name), 0), 1)
  Wrong:     COUNT(...) / COUNT(...) * 100  ← always 0 due to integer division

SQL GENERATION RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. SELECT only. Never generate INSERT, UPDATE, DELETE, DROP, ALTER, TRUNCATE, or any write operation.
2. Always use LOWER() for name comparisons: WHERE LOWER(object_name) = LOWER('WApp01').
3. protected_vms has duplicate rows — always COUNT(DISTINCT object_name) and SELECT DISTINCT object_name when querying that table.
   For failed_jobs_daily: do NOT use SELECT DISTINCT. Every row is a meaningful failure event — return all rows so the user sees every failure.
4. DISTINCT + ORDER BY constraint: when using SELECT DISTINCT, every column in ORDER BY must also appear in the SELECT list. If you need to ORDER BY a column not in the SELECT, remove DISTINCT instead.
5. status column exists only in job_sessions. Do not reference status in any other table.
   failed_object_name exists only in failed_jobs_daily. Do NOT use failed_object_name in job_sessions.
6. Date filters: CURRENT_DATE = today only, CURRENT_DATE-1 = last 24 hours (yesterday + today), CURRENT_DATE-7 = last 7 days.
   "Recent" always means last 24 hours — use backup_date >= CURRENT_DATE - 1.
   "Today" means backup_date = CURRENT_DATE.
   Never use just CURRENT_DATE when the user says "recent" or "latest".
7. For failure queries: always include failed_object_name AND failure_message so the result is actionable.
8. No artificial LIMIT on list or detail queries. Add LIMIT only if the user asks for a specific number of results.
9. Counting failures: always use COUNT(DISTINCT failed_object_name) for objects and COUNT(DISTINCT job_name) for jobs — never COUNT(*) on failed_jobs_daily as the same object can appear on multiple dates.
9. When joining failed_jobs_daily with protected_vms to get object_type, join on LOWER(p.object_name) = LOWER(f.failed_object_name).
